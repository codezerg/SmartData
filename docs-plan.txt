SmartData Documentation Plan
============================

Goal: MS-docs-style IA (pre-2019 MSDN era) — task-oriented, layered, heavily
cross-linked. NOT a blind reference dump per project.

Docs live in the Astro Starlight site at site/src/content/docs/ and deploy to
https://smartdata-apis.netlify.app/. Raw long-form source material stays in
docs/*.md at the repo root — treat these as the pool, not the published surface.

Design Principles
-----------------

1. Task-first, reference second.
   Sidebar runs overview → get-started → fundamentals → how-to → tutorials →
   reference → samples. Reference is the last section, not the first.

2. Every how-to answers one question.
   Titled "How to X". Scannable in 60 seconds. Paste-ready code sample.

3. Fundamentals explain the mental model once.
   Other pages link back instead of re-explaining concepts.

4. Reference pages are thin.
   API surface — types, signatures, one-line descriptions. Readers arrive
   knowing what they want; they don't read top-to-bottom.

5. Cross-linking is heavy.
   A how-to on scheduling links to fundamentals/scheduling.md and the
   SysSchedule reference row.

6. No project-shaped docs at the top level.
   Users don't care if something lives in SmartData.Server vs SmartData.Core.
   Reference is the only place where project boundaries matter.


Decisions (previously open questions)
-------------------------------------

- Overview/Get-started split — KEPT separate. Overview is conceptual ("what
  is this, how does it fit together"), Get Started is action ("install,
  write your first procedure").

- Fundamentals (not Concepts) — using Fundamentals throughout.

- Best-practices section — SKIPPED. Add organically once patterns earn their
  place. Empty/aspirational sections rot.

- Reference grouping — one page per project. Matches the code, easier to
  maintain, unambiguous for consumers who've already picked a package.

- System procedures — grouped by concern (schema / backup / scheduling /
  tracking / logs / users / storage) inside a single reference page. Easier
  to scan than splitting into seven micro-pages.

Out of Scope (for now)
----------------------

- Versioned docs (multiple side-by-side versions)
- API-extracted XML-doc reference pages (Docfx or similar) — compliant XML
  comments aren't universal yet
- Algolia or external search — Starlight's built-in Pagefind covers it
- i18n
