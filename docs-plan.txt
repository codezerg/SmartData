SmartData Documentation Plan
============================

Goal: MS-docs-style IA (pre-2019 MSDN era) — task-oriented, layered, heavily
cross-linked. NOT a blind reference dump per project.

Design Principles
-----------------

1. Task-first, reference second.
   Sidebar opens with overview/get-started/how-to. Reference is the last
   section, not the first.

2. Every how-to answers one question.
   Titled "How to X". Scannable in 60 seconds. Code sample you can paste.

3. Fundamentals explain the mental model once.
   Other pages link to them instead of re-explaining concepts.

4. Reference pages are thin.
   Just the API surface — types, signatures, 1-line descriptions. Readers
   arrive here knowing what they want; they don't read top-to-bottom.

5. Cross-linking is heavy.
   A how-to on scheduling links to the scheduling concept page and the
   SysSchedule reference row.

6. No project-shaped docs at the top level.
   Users don't care if something lives in SmartData.Server vs SmartData.Core
   — they care about the task. Reference is the only place where project
   boundaries matter.


Open Questions (decide before writing)
--------------------------------------

- Overview/Get-started split — one section or two?

- "Fundamentals" vs "Concepts" — naming only.

- Best-practices section? (e.g. "When to use a procedure vs a raw query")
  Valuable if written well, dead weight if half-written.

- Reference grouping:
  (a) one page per project (current plan, easier to maintain), or
  (b) reorganize by topic across assemblies (more MS-docs-like).

- System procedures: one big page, or grouped by concern
  (schema, backup, scheduling, tracking, logs)?
